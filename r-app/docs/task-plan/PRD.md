# PRD：统计页面与仪表盘改造

> 状态：ready-for-agent
> 来源：基于 `new_features_task.txt` 与本次需求对齐讨论综合而成
> 适用范围：tauri-gateway（r-app，React 19 + Tauri/Rust + SQLite）

## Problem Statement

当前统计页面只展示按"今日/昨日/本周/本月"的聚合数字与一个简单的按月归档面板，用户无法看到**单条请求的实时明细**（请求时间、入站/出站、响应状态码、Token 明细），也看不到**缓存创建/缓存读取**这类对成本影响很大的 Token 维度。

仪表盘底部只是把端点和脱敏密钥罗列出来，信息价值低；服务状态区域用一张独立卡片只显示"端点数量"，浪费版面；用户启动本地代理后缺乏直观的"正在工作"反馈。

此外，用户在本机使用 Claude Code 与 Codex 产生的 Token 用量散落在本地 JSONL 会话文件中，应用内完全没有汇总入口，用户无法一眼看清自己每天/每个模型用了多少。

## Solution

把"统计"与"仪表盘"两个页面围绕**端点请求明细**和**本机工具用量**两条数据线重构：

1. **统计页**顶部新增一级 Tab：`端点统计` 与 `用量统计`。
   - `端点统计` 上半保留周期聚合，端点表新增"缓存创建/缓存读取"两列；原"归档"改为"历史记录"弹窗，分页展示按日明细并支持按行/按天删除。
   - `端点统计` 下半改为**实时端点请求监控**：逐条展示请求时间、入站请求、出站请求、响应状态码，Token 明细以悬停图标展开（输入/输出/缓存创建/缓存读取/合计）。支持按时间段查询（ranged 模式）。
   - `用量统计` 读取本机 Claude Code 与 Codex 的会话日志，增量同步进库后，按应用/模型/天汇总 Token（MVP：不计算成本）。

2. **仪表盘**重构首屏：
   - 第一张卡片改为"左 3 右 2"宽度：左侧展示启用端点列表与当前工作端点；右侧展示本地代理信息 + 启停开关，端口文字可点击跳转设置；代理启动后该卡片呈现"海水涨潮"动效。
   - 删除独立的"端点数量"卡片，把服务状态信息并入第一张卡片；原"端点数量"位置改为请求数/失败数/Token 的实时统计。
   - 底部"端点+密钥"卡片替换为**实时请求监控**（live 模式，事件驱动实时追加，分页展示），并展示启用端点的别名（用端点 `name`）。

3. **复用**：实时请求监控同时服务于统计页（按时间段）与仪表盘（实时），后端共享一套明细数据源，前端抽象为一个支持 `live | ranged` 两种模式的监控组件。

## User Stories

1. 作为网关使用者，我希望统计页顶部能在"端点统计"和"用量统计"之间切换，以便分别查看网关流量和本机工具用量。
2. 作为使用者，我希望端点统计保留"今日/昨日/本周/本月"周期切换，以便对比不同时间窗口的总量。
3. 作为使用者，我希望端点统计的汇总卡片继续展示请求数/错误数/输入 Token/输出 Token，并在"今日"显示环比趋势，以便快速感知变化。
4. 作为关注成本的使用者，我希望端点统计表格新增"缓存创建"和"缓存读取"两列，以便看清每个端点的缓存 Token 消耗。
5. 作为使用者，我希望系统在解析上游响应时正确记录 `cache_creation_input_tokens` 与 `cache_read_input_tokens`，以保证缓存 Token 数据准确。
6. 作为使用者，我希望原"归档"改为"历史记录"，通过弹窗查看历史聚合明细，以便不占用主页面版面。
7. 作为使用者，我希望历史记录弹窗以分页方式展示按日明细（端点×日的聚合行），以便翻阅长期数据。
8. 作为使用者，我希望能在历史记录里删除单行（某端点某日）或整天的记录，以便清理无用数据。
9. 作为使用者，我希望端点统计下半部分是实时请求监控，逐条展示每个请求，以便观察网关实时活动。
10. 作为使用者，我希望每条请求展示请求时间，以便定位某次调用。
11. 作为使用者，我希望每条请求展示入站请求信息（入站格式/端点），以便知道请求来自哪种客户端协议。
12. 作为使用者，我希望每条请求展示出站请求信息（实际转发的上游端点/URL），以便确认走了哪个上游。
13. 作为使用者，我希望每条请求展示响应状态码，以便快速识别失败请求。
14. 作为使用者，我希望每条请求有一个 Token 图标，悬停后展开 Token 明细（输入/输出/缓存创建/缓存读取/合计），以便不展开时保持表格清爽。
15. 作为使用者，我希望端点统计下半的监控支持按时间段查询历史请求，以便回溯过去某段时间的请求。
16. 作为使用者，我希望监控列表能分页，以便在请求量很大时浏览。
17. 作为使用者，我希望明细数据自动保留最近 90 天并清理更早的数据，以便磁盘占用可控。
18. 作为 Claude Code 用户，我希望用量统计能读取本机 `~/.claude/projects` 下的会话日志，统计我的 Token 用量。
19. 作为 Codex 用户，我希望用量统计能读取本机 `~/.codex/sessions` 下的会话日志，统计我的 Token 用量。
20. 作为使用者，我希望用量统计能区分 Claude Code 与 Codex 两个来源分别展示，以便对比。
21. 作为使用者，我希望用量统计按模型汇总 Token，以便看清各模型用量分布。
22. 作为使用者，我希望用量统计按天汇总 Token，以便看到每日趋势数据。
23. 作为使用者，我希望用量统计展示总量卡片（总请求/总输入/总输出/缓存 Token），以便一眼看到整体规模。
24. 作为使用者，我希望用量数据增量同步（只解析新增内容），以便重复打开时快速且不重复计数。
25. 作为使用者，我希望用量统计页有手动刷新按钮，以便在使用工具后立即更新数据。
26. 作为使用者，我希望仪表盘第一张卡片左侧展示启用端点列表与当前工作端点，以便集中查看端点状态。
27. 作为使用者，我希望第一张卡片右侧展示本地代理信息与启停开关，以便在仪表盘直接控制代理。
28. 作为使用者，我希望点击代理端口文字能跳转到设置页，以便快速修改端口配置。
29. 作为使用者，我希望代理启动后第一张卡片有"海水涨潮"动效，以便直观感知代理正在运行。
30. 作为使用者，我希望仪表盘不再有独立的"端点数量"卡片，服务状态信息并入第一张卡片，以便版面更紧凑。
31. 作为使用者，我希望原"端点数量"卡片位置改为请求数/失败数/Token 的实时统计，并实时更新，以便在仪表盘掌握当前流量。
32. 作为使用者，我希望仪表盘底部不再罗列端点和密钥，改为实时请求监控（live 模式），以便看到最新请求。
33. 作为使用者，我希望仪表盘的实时监控能分页，以便浏览较多请求。
34. 作为使用者，我希望仪表盘实时监控展示启用端点的别名（端点 name），以便辨认端点。
35. 作为使用者，我希望仪表盘实时监控在新请求发生时自动追加（事件驱动），无需手动刷新。
36. 作为开发者，我希望统计页与仪表盘的实时监控复用同一后端数据源与同一前端组件（区分 live/ranged 模式），以便降低重复实现与维护成本。

## Implementation Decisions

### 总体分层
- 划分为四个工作包：WP1 数据层（后端地基）、WP2 统计页、WP3 用量统计、WP4 仪表盘。
- WP1 为关键路径，WP2/WP4 的监控功能依赖它；WP3 与 WP1 解耦，可并行。

### WP1 数据层（后端 + 类型）
- **缓存 Token 维度**：在 `daily_stats` 表新增 `cache_creation_tokens`、`cache_read_tokens` 两列，通过新增一条版本化迁移（v3）追加，沿用现有 `schema_version` 增量迁移机制（不破坏既有 v1/v2）。
- **上游响应解析**：在 Token 解析逻辑中补充对 Claude `cache_creation_input_tokens` / `cache_read_input_tokens` 的读取，覆盖非流式与流式累加两条路径；OpenAI 入站若无对应字段则记 0。
- **聚合器与仓储**：内存增量结构、UPSERT 写入、周期查询、月度/历史查询同步带上两个缓存字段；相关模型结构体（单日统计、端点聚合、周期聚合）扩展两个字段。现有 `get_stats` 返回结构与 `stats-updated` 事件保持向后兼容（仅增字段）。
- **请求明细表 `request_logs`（新增）**：每条请求落一行。字段约定：
  - `id`（自增）、`ts`（请求/落库时间戳，毫秒或 RFC3339）、`endpoint_name`、`inbound_format`（入站协议/格式）、`upstream_url`（实际转发上游）、`status_code`、`is_error`、`input_tokens`、`output_tokens`、`cache_creation_tokens`、`cache_read_tokens`、`model`（可空）、`duration_ms`（可空）、`device_id`。
  - 保留策略：仅保留最近 **90 天**，按天清理（插入时或应用启动时删除超期行）。
- **采集点（双写）**：在代理转发的统一记录汇聚点同时写两处——继续累加 `daily_stats`（聚合，供端点统计上半与历史记录），并向 `request_logs` 追加一行明细（供监控）。需要把 `model`、`status_code`、时间戳、入站格式、出站 URL 透传到该汇聚点（这些信息在转发入口处可得，但当前未传递到记录处）。
- **流式请求**：在流结束、拿到最终 Token 后再落 `request_logs` 行；`duration_ms` 取请求开始到结束；first-token 暂不采集。
- **命令与事件契约（新增/调整）**：
  - `get_request_logs(filter, page, page_size) -> { items: RequestLog[], total }`，`filter` 支持时间段（start/end）与可选端点过滤。
  - `request-logged` 应用→前端事件，payload 为新落库的单条 `RequestLog`，供 live 模式实时追加。
  - `get_stats_history(page, page_size) -> { items: DailyStat[], total }`：跨全时间分页读取 `daily_stats`（替代原按月接口的列表能力）。
  - `delete_daily_stat(endpoint_name, date)` 与 `delete_stats_by_date(date)`：分别支持按行与按整天删除。
  - 原按月归档命令（按月查询/删除）在历史记录改造后由上述分页/删除命令取代。

### WP2 统计页（前端）
- 顶部一级 `Tabs`：`端点统计` | `用量统计`；原周期 `Tabs` 降为 `端点统计` 内的子级。
- 端点统计表新增"缓存创建/缓存读取"两列；汇总卡片维持四张不变（缓存数据放表格列，不再加大卡片）。
- "历史记录"用 `Dialog` 承载：内部为分页表格 + 行内删除/按天删除按钮；数据走 `get_stats_history` 与删除命令；操作后失效相关查询缓存。
- 端点统计下半接入共享监控组件，`mode="ranged"`，提供时间段选择，走 `get_request_logs` 分页查询。
- 需要新增一个通用分页组件（当前代码库没有）。

### WP3 用量统计（后端 + 前端，MVP）
- **数据来源**：Claude Code 读 `~/.claude/projects/<project>/*.jsonl`（含子代理目录），取 `type=="assistant"` 行的 `message.usage`（input/output/cache_read/cache_creation）；Codex 读 `~/.codex/sessions/YYYY/MM/DD/*.jsonl` 与 `archived_sessions/*.jsonl`，取 `token_count` 事件的 `total_token_usage`（累计值，按相邻差取增量）。
- **增量同步进 SQLite**：新增用量表 + 同步进度表（按文件 mtime + 行偏移记录已解析位置），仅解析新增内容；Claude 按 `message.id` 去重，Codex 用累计差并对负值/缓存超额做钳制。用量表与 `request_logs` **相互独立、不做跨源去重**（两者衡量不同对象：网关转发流量 vs 本机工具会话）。
- **聚合维度**：按 `app_type`（claude/codex）、`model`、`date` 聚合 Token 总量与请求计数。
- **命令契约**：
  - `sync_session_usage() -> { imported, skipped, files_scanned, errors }`：触发一次增量同步。
  - `get_usage_summary(start?, end?, app_type?) -> UsageSummary`（总请求、各类 Token 总量）。
  - `get_usage_by_model(start?, end?, app_type?) -> ModelUsage[]`。
  - `get_usage_by_day(start?, end?, app_type?) -> DailyUsage[]`。
- **触发方式**：进入"用量统计"Tab 时先调用一次 `sync_session_usage` 再查询；页面提供手动刷新按钮。MVP 不做文件监听、不做实时事件。
- **展示**：总量卡片 + 按模型表 + 按天表；提供 app_type 过滤（全部/Claude Code/Codex）。**不计算成本、不做定价表、不做大趋势图。**

### WP4 仪表盘（前端）
- 第一张卡片改为左 3 / 右 2 栅格：
  - 左：启用端点列表 + 当前工作端点高亮（数据来自现有 health/proxy 状态）。
  - 右：本地代理信息 + 启停 `Switch`；端口文字渲染为可点击元素，点击路由到设置页；代理 `running` 时叠加"海水涨潮"动效（用已引入的 `motion` 实现，水位/波浪上升）。
- 删除"端点数量"独立卡片；服务状态（运行中/已停止）并入第一张卡片；原位置改为请求数/失败数/Token 三个实时数字，订阅已有 `stats-updated` 事件并读取"今日"周期数据更新。
- 底部"端点+密钥"卡片替换为共享监控组件，`mode="live"`：订阅 `request-logged` 事件实时追加、支持分页；行内展示启用端点别名（端点 `name`）。

### Req9 共享组件
- 后端：`request_logs` 表 + `get_request_logs` + `request-logged` 事件构成共享数据源。
- 前端：抽象 `RequestMonitor` 组件，`mode` 取 `live`（事件追加、聚焦最近）或 `ranged`（时间段 + 分页查询）；列与 Token 悬停明细两模式共用，差异仅在数据获取与刷新策略。

## Testing Decisions

测试只验证外部行为（输入→输出 / 落库结果 / 渲染结果），不绑定内部实现细节。沿用既有测试种子（seam），优先在现有最高层 seam 加测，仅在必要处新增。

- **缓存 Token 解析（后端）**：在 Token 解析模块的单元测试中补充用例——给定含 `cache_creation_input_tokens`/`cache_read_input_tokens` 的 Claude 响应（非流式与流式分片），断言解析出的缓存 Token 正确；OpenAI 无字段时为 0。先例：该模块已有 6 个解析测试。
- **聚合与仓储（后端）**：在统计仓储测试中补充缓存列的 UPSERT 累加、周期查询、`get_stats_history` 分页、`delete_daily_stat`/`delete_stats_by_date` 行为。先例：统计仓储与端点仓储已有针对内存 SQLite 的测试。
- **迁移（后端）**：在迁移测试中断言 v3 迁移后 `daily_stats` 含两个新列、`request_logs` 表建成且旧库可平滑升级。先例：迁移模块已有 2 个测试。
- **请求明细仓储（后端，新增 seam）**：新增请求明细仓储测试，覆盖插入、按时间段分页查询、90 天清理删除超期行。仿照统计仓储测试用内存 SQLite。
- **用量同步与聚合（后端，新增 seam）**：用 fixture JSONL（Claude assistant 行 / Codex token_count 累计行）测试增量解析、去重/取差、按 app/model/day 聚合结果；以及二次同步对同一文件不重复计数（偏移推进）。
- **前端组件（vitest + testing-library）**：
  - `RequestMonitor`：ranged 模式渲染分页数据、Token 悬停展开明细；live 模式收到 `request-logged` 事件后追加新行。
  - 历史记录弹窗：分页、删除按钮触发对应命令并刷新。
  - 用量统计：总量卡片、按模型/按天表在给定数据下正确渲染，app_type 过滤生效。
  - 通用分页组件：页码切换回调正确。
  - 先例：`src/__tests__` 下已有组件/逻辑测试（如 TestBadge、filters）。

## Out of Scope

- 用量统计的**成本/费用计算**、可编辑**定价表**、以及大型**趋势折线图**（本期 MVP 不做，仅 Token 数量维度）。
- 端点**独立 alias 字段**：仪表盘"别名"直接用端点 `name`，不新增字段、不改端点 CRUD。
- 用量数据的**文件监听 / 实时推送**：MVP 仅在进入 Tab 时同步 + 手动刷新。
- 用量表与网关 `request_logs` 的**跨源去重**：两者独立。
- 监控明细的**手动逐条删除**：明细仅靠 90 天自动清理；手动删除只针对历史记录（daily_stats 聚合）。
- 请求监控的 **first-token 延迟**采集（仅采集总耗时）。
- 多设备用量聚合、导出、告警等增强能力。

## Further Notes

- 现有 `get_stats` / `stats-updated` / 趋势计算保持向后兼容，改造仅做"增列"，不改变既有返回语义，最大限度保护调用链。
- `daily_stats`（聚合）与 `request_logs`（明细）是两条并行数据路径：上半端点统计与历史记录读聚合，下半监控读明细，互不影响，降低破坏既有功能的风险。
- 构建顺序建议：WP1 先行 → WP2 监控 / WP4 监控 / WP4 布局并行 → WP3 全程可并行 → 海水涨潮动效最后打磨。
- 端口在前端已是动态 `status.port`，并非写死 3000；"点击端口字样跳设置"对应把该文字做成可点击的路由入口。
