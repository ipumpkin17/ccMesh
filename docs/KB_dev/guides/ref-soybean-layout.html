01 整体目录结构总览

克隆后 src/ 目录呈现出清晰的职责分层，views 和 components 是两个完全不同性质的层：

src/

│ ├── _builtin/ # 框架内置页（login / 403 / 404 / 500）
│ │ ├── login/
│ │ │ ├── index.vue # 页面入口，负责组装
│ │ │ └── modules/ # 页面私有子模块

│ └── home/
│ ├── index.vue # 页面入口
│ └── modules/

核心原则： views/ 是路由的归属地，组件只服务于所在页面；components/ 是真正的"公共资产"，任何页面都能用。两者不混放。

02 views 的设计模式：页面 + modules 双层结构

每个路由页面都遵循相同的两层结构：

第一层 — index.vue

页面的"导演"。只负责布局编排：从 modules/ 导入子模块，用 Grid/Space 组织位置关系，从 store 取顶层数据（如 isMobile），不包含具体业务逻辑。

第二层 — modules/*.vue

页面的"演员"。每个文件是一个有明确业务含义的独立区块，自带数据获取、状态、渲染逻辑。私有于该页面，不对外暴露。

home/index.vue 的职责（纯组装）

// home/index.vue — 只做两件事：导入子模块 + 排版
import HeaderBanner from './modules/header-banner.vue';
import CardData     from './modules/card-data.vue';
import LineChart    from './modules/line-chart.vue';
import PieChart     from './modules/pie-chart.vue';
import ProjectNews  from './modules/project-news.vue';
import CreativityBanner from './modules/creativity-banner.vue';

// 模板只写布局，没有任何业务数据
<NSpace vertical :size="16">
  <HeaderBanner />
  <CardData />
  <NGrid>
    <NGi><LineChart /></NGi>
    <NGi><PieChart /></NGi>
  </NGrid>
</NSpace>vue

login/index.vue 的"动态模块"模式

Login 页更进一步：将所有子模块放入 moduleMap，通过 <component :is="..."> 动态渲染，实现"同 URL、不同视图"的切换（密码登录 → 注册 → 找回密码），避免路由跳转。

const moduleMap: Record<LoginModule, { label; component }> = {
  'pwd-login':  { label: ..., component: PwdLogin  },
  'code-login': { label: ..., component: CodeLogin },
  'register':   { label: ..., component: Register  },
  'reset-pwd':  { label: ..., component: ResetPwd  },
};

// 模板中动态渲染当前模块
<component :is="activeModule.component" />vue
规律总结： modules/ 里的组件是"页面私有组件"，和 components/ 里的"全局组件"本质区别在于——能否被其他页面复用。不能复用的放 modules/，可复用的才提升到 components/。

common 组件的特征

以 full-screen.vue 为例，完全无副作用，Props 只有一个布尔值，纯 UI 呈现：

// full-screen.vue — 只接收状态，不产生副作用
interface Props { full?: boolean }
defineProps<Props>();

// 模板：根据 props 显示不同图标
<ButtonIcon :tooltip-content="full ? '退出' : '全屏'">
  <icon-gridicons-fullscreen-exit v-if="full" />
  <icon-gridicons-fullscreen v-else />

advanced 组件的特征

// table-header-operation.vue — 有逻辑 + 有通信
interface Emits {
  (e: 'add'): void;
  (e: 'delete'): void;
  (e: 'refresh'): void;
}
const columns = defineModel<TableColumnCheck[]>('columns', { default: () => [] });vue

04 命名风格：文件、组件、Props

文件名：全部 kebab-case（短横线小写）

card-data.vue header-banner.vue pwd-login.vue table-column-setting.vue dark-mode-container.vue

无一例外，所有 .vue 文件使用 kebab-case，与 HTML 标签风格一致，避免大小写在不同操作系统文件系统上的歧义。

组件 name：PascalCase（大驼峰）

每个组件都通过 defineOptions({ name: '...' }) 显式声明 PascalCase 名称，且严格与文件名对应（kebab ↔ pascal）：

文件名defineOptions name模板中使用
card-data.vueCardData<CardData />
header-banner.vueHeaderBanner<HeaderBanner />
pwd-login.vuePwdLogin<PwdLogin />
system-logo.vueSystemLogo<SystemLogo />
svg-icon.vueSvgIcon<SvgIcon />
count-to.vueCountTo<CountTo />
button-icon.vueButtonIcon<ButtonIcon />

Props 命名：camelCase 定义，kebab-case 使用

// 定义：camelCase
interface Props {
  themeSchema: UnionKey.ThemeScheme;   // themeSchema
  showTooltip?: boolean;               // showTooltip
  tooltipPlacement?: PopoverPlacement; // tooltipPlacement
}

// 使用：kebab-case（Vue 自动转换）
<ThemeSchemaSwitch
  :theme-schema="themeStore.themeScheme"
  :show-tooltip="false"
/>vue

emit 事件：camelCase

interface Emits {
  (e: 'add'): void;
  (e: 'delete'): void;
  (e: 'changeLocale', lang: string): void;  // camelCase
}ts
命名规律： 文件 kebab-case → 组件名 PascalCase → Props camelCase（模板中 kebab）→ Emit camelCase。整条链路非常一致，Vue 官方推荐风格。

05 组件导入方式：自动导入 vs 显式导入

Soybean Admin 用了两种截然不同的导入策略，分别对应不同的组件层级：

策略 A — 自动导入（unplugin-vue-components）

components/ 下的所有组件 + NaiveUI 组件 + Iconify 图标，由 Vite 插件在编译时扫描并注册为全局组件。模板中直接使用，无需 import。

策略 B — 显式 import（相对路径）

views/*/modules/ 下的页面私有组件，必须在 index.vue 的 <script setup> 中用相对路径显式导入。体现"私有性"。

自动导入配置（build/plugins/unplugin.ts）

Components({
  dts: 'src/typings/components.d.ts',  // 生成类型声明
  resolvers: [
    NaiveUiResolver(),               // NaiveUI 组件自动导入
    IconsResolver({
      customCollections: [collectionName],
      componentPrefix: VITE_ICON_PREFIX  // 图标组件自动导入
    })
  ]
})ts

自动生成的 components.d.ts 会声明所有全局组件类型，让 TypeScript 知道 <SvgIcon>、<CountTo>、<NButton> 这些组件的存在。

两种导入在代码中的对比

// ─ home/index.vue ────────────────────────────────────────
import HeaderBanner from './modules/header-banner.vue'; // ← 显式导入（私有模块）
import CardData     from './modules/card-data.vue';     // ← 显式导入（私有模块）

// 模板中可以直接用（自动导入，无需 import）：
<SvgIcon />        // components/custom/svg-icon.vue
<CountTo />        // components/custom/count-to.vue
<NCard />          // NaiveUI 组件
<NGrid />          // NaiveUI 组件
<SoybeanAvatar />  // components/custom/soybean-avatar.vuevue
关键区分： 是否需要写 import 就是区分"全局组件"和"私有模块"的直接标志。需要显式 import 的，是只属于该页面的 modules；不需要 import 就能用的，是全局注册的 components。

07 迁移到 React：目录组织如何重构

Vue 的 views/ + modules/ + components/ 三层结构在 React 中需要做几处关键转换：

差异对照表

Vue 版（Soybean）React 版对应做法说明
views/*/index.vue pages/*/index.tsx 目录名从 views → pages（React 生态更通用）
views/*/modules/*.vue pages/*/_components/*.tsx 用 _ 前缀或 components/ 子目录表示私有，Next.js 风格
components/common/ components/common/ 结构相同，扩展名 .tsx
components/custom/ components/ui/ 或 components/shared/ shadcn/ui 风格叫 ui/，通用增强叫 shared/
components/advanced/ components/business/ 或 features/*/components/ 如采用 Feature Sliced Design，拆到 features/ 下
defineOptions({ name }) 函数名即组件名（天然具名） React 函数组件名 = displayName，无需额外声明
unplugin-vue-components 自动导入 全部显式 import（无自动导入） React 生态无等价方案，靠 barrel export (index.ts)
kebab-case 文件名 PascalCase 文件名 React 约定：组件文件用 PascalCase

React 没有自动导入，但可以用 index.ts 聚合导出，模拟"按需使用"的便利性：

export { default as AppProvider     } from './AppProvider';

// 使用方（页面）
import { FullScreen, SystemLogo } from '@/components/common';ts

home/index.tsx — 页面组装层

Vue 原版
import HeaderBanner from
  './modules/header-banner.vue';
import CardData from
  './modules/card-data.vue';

const appStore = useAppStore();
const gap = computed(() =>
  appStore.isMobile ? 0 : 16);

import HeaderBanner from
  './_components/HeaderBanner';
import CardData from
  './_components/CardData';

const isMobile = useAppStore(
  s => s.isMobile);
const gap = isMobile ? 0 : 16;

CardData.tsx — 页面私有模块（React 版）

// pages/home/_components/CardData.tsx
// ← 文件名 PascalCase（React 约定）
// ← 放在 _components/ 下（私有于 home 页面）

interface CardItem {
  key: string;
  title: string;
  value: number;
  color: { start: string; end: string };
  icon: string;
}

const CardData = () => {
  const { t } = useTranslation();
  // React 函数名本身就是 displayName，无需 defineOptions

  const cardData: CardItem[] = [
    { key: 'visitCount', title: t('visitCount'), ... }
  ];

  return (
    <Card>
      {cardData.map(item => (
        <div key={item.key}
          style={{background: gradient(item.color)}}>
          <CountTo endValue={item.value} />
        </div>
      ))}
    </Card>
  );
};

export default CardData;tsx

SvgIcon.tsx — 全局 UI 组件（React 版）

// components/ui/SvgIcon.tsx
// ← 放在 components/ui/ 下（全局共享）
// ← 所有用到的地方都要显式 import（无自动导入）

interface Props {
  icon?: string;        // Iconify 图标名
  localIcon?: string;   // 本地 SVG 图标
  className?: string;
}

const SvgIcon = ({ icon, localIcon, className }: Props) => {
  if (localIcon) {
    return <svg className={className}>...</svg>;
  }
  return <Icon icon={icon!} className={className} />;
};

export default SvgIcon;tsx

命名规则总结（Vue → React）

维度Vue (Soybean)React 对应做法
文件名kebab-case（card-data.vue）PascalCase（CardData.tsx）
组件显示名defineOptions({ name: 'CardData' })函数名 const CardData = () => ...
页面私有模块目录views/page/modules/pages/page/_components/
全局组件导入自动导入（unplugin）显式 import + barrel index.ts
Props 定义interface Props + defineProps<Props>()interface Props + 函数参数解构
Props 使用kebab-case（:show-tooltip）camelCase（showTooltip={false}）
Emit 事件defineEmits + 字符串事件名onXxx 回调 Props（onAdd, onDelete）
双向绑定defineModel（v-model:columns）value + onChange Props 模式
状态管理Pinia useXxxStore()Zustand useXxxStore() 或 Redux

核心结论： Soybean 的架构思路（页面入口只做组装、业务拆进 modules、全局组件三级分类）在 React 中完全可以复现。最大差异是：① 文件名从 kebab 变 Pascal；② 全局组件从自动导入变为 barrel 手动聚合；③ Vue 的 defineOptions/defineModel 等 API 在 React 里由函数名和 Props 约定自然替代；④ 目录名惯例从 views → pages，modules → _components。
