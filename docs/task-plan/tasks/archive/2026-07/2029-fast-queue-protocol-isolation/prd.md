# 快速队列协议隔离

## 问题

存在任意快速端点时，代理先全局收窄到快速队列，再识别请求协议。只配置一个 `codex` 快速端点时，Claude Code 也会被迫使用该端点，导致 `/v1/messages` 被发往 Responses 上游并出现模型不存在等错误。

## 决策

- 先识别入站协议并筛选可透传或可转换的端点
- 仅在兼容端点内部启用快速队列优先级
- 当前协议没有兼容快速端点时，回退到该协议的全部普通启用端点
- Claude 入站明确排除原生 Responses 端点
- 不同入站协议分别维护轮换状态，使用稳定端点 ID 而非候选数组下标
- 不新增快速队列字段，不改变现有编辑界面和数据库结构

## 验收标准

- Codex 快速端点不影响 Claude Code 的候选端点
- Claude 快速端点不影响 Codex Responses 的候选端点
- 同一协议兼容的快速端点仍按 `fast_sort_order` 轮询
- Claude 请求永不直接路由到原生 Responses 端点
- 手动切换在协议过滤、模型过滤或候选重排后仍指向目标端点
