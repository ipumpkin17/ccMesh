## 熔断器 HTTP 状态码分类原则

> 一句话结论：4xx 请求侧错误中性不计熔断；403 与 429/5xx 一样计失败

**你会遇到这个问题的场景**
多上游网关的熔断/轮换逻辑用 HTTP status 决定是否标记端点不健康、是否切换渠道。400/401/422 与 403/429/502 若混为一类，行为会偏离预期。

**为什么会出错**
400/401 常是**客户端请求**问题（坏 JSON、坏 key），不应因一次错误就把上游标死。403 在部分聚合层表示**渠道权限/配额/UA 限制**，重试同一端点无意义，若不累计失败会无限打无效渠道。分类函数（如 `categorize_status`）影响整条 forward → breaker → UI 链路。

**正确做法**
- **中性（不计熔断）**：400、401、422 等明确请求格式/鉴权客户端侧问题
- **计失败/可轮换**：403、429、5xx、连接超时等上游或配额侧问题
- 403 从「client error 豁免列表」中移除，与 429 同等对待
- 分类表写入单测；改一处，forward 与 breaker 共用

**反例**
❌ 错误：403 与 400 同属 client error，永不 trip breaker  
✅ 正确：403 累计失败，触发轮换或熔断
