## 参考实现不可移植时的选型策略

> 一句话结论：私有 crate 不假移植，改自研 + 外部映射表校对

**你会遇到这个问题的场景**
选定某开源/商业项目为参考，准备移植协议转换、配置 IO 等模块。调研发现核心逻辑在私有 crate、或未开源的分支里。

**为什么会出错**
仅凭公开 README 或部分文件「猜」私有模块行为，会做出与参考不一致的半套实现，且无法 diff 验证。强行依赖不可获得的 crate 会阻塞构建。

**正确做法**
- research 阶段给每个依赖模块打三档：**可整体移植** / **仅架构蓝图** / **不可用**
- 不可用档：放弃移植，列自研范围 + 外部可读的第二参考（如 Go 版 bridge 的字段表）；例：参考实现的 `canonicalize_tool_arguments*` 在私有 crate → 自研 `canonical_json_string` + 外部映射表单测校对
- 可移植档：带 file:line 锚点整文件移植，而非按摘要重写
- 验收：逐字段映射表 + 单测/fixture，而非「感觉差不多」

**反例**
❌ 错误：EchoBird 核心在 `echobird_core` 私有仓 → 仍按公开 wrapper 猜转换逻辑  
✅ 正确：标记不可用，自研转换器 + moon-bridge 映射表逐条校对
