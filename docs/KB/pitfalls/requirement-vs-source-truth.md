## 二手需求须对照真实源码再设计

> 一句话结论：移植或复刻前，以参考项目源码为准，勿信需求摘要

**你会遇到这个问题的场景**
从其他项目、文档摘要或口头描述复刻功能时，直接按需求文档设计接口与模块边界，上线后发现参考实现根本没有文档里声称的能力，或行为完全不同。

**为什么会出错**
需求摘要往往在传递过程中被简化、误读或过时。撰写者可能把「期望行为」当成「已有行为」，或混淆了不同版本/分支。没有 file:line 级对照，设计会基于虚构前提展开，浪费整轮实现。

**正确做法**
- 动手前在参考项目 grep 验证每个声称的能力是否存在
- 写 research 文档时建立「摘要说法 vs 源码真相」对照表，标注证据行号
- PRD 显式记录差异与取舍（做 / 不做 / 改做）
- 仅移植有源码锚点的逻辑；无锚点的能力标记为待验证或放弃

**反例**
❌ 错误：摘要写「Rectifier 会剥离空 tool 参数」→ 直接实现剥离逻辑  
✅ 正确：grep 发现 rectifier 只处理 thinking 签名 → 空参数改在请求构建期主动 canonicalize
