## 上游模型拉取：鉴权聚合 + URL 路径候选

> 一句话结论：先轮换鉴权头，再剥离兼容子路径试候选 URL

**你会遇到这个问题的场景**
用户在端点表单填了 API URL 和协议类型，点「刷新模型列表」。实际 URL 可能多/少一段 `/v1`、或协议类型选错导致鉴权方式不匹配，但同一物理服务仍可用另一种 auth 或路径访问 `/models`。

**为什么会出错**
只按用户所选 transformer 发一次 GET `/v1/models`，失败即报「无法获取」。用户常把 OpenAI 与 Anthropic 鉴权、或 Chat 与 Responses base path 搞混。若把此重策略放进 bulk 聚合或 forward 热路径，会拖慢每次请求。

**正确做法**
- **第一步**：按所选 protocol 的鉴权头发请求；失败换另一种鉴权**同一 URL** 重试
- **第二步**：仍失败则根据已知兼容规则生成 URL 候选（如剥/补 `/v1`、去尾斜杠），逐个试
- 任一步成功即返回，不继续浪费
- 此逻辑**仅用于**表单刷新 / models_probe 等显式探测，不与 forward 端点轮换或 bulk 聚合混用
- 维护候选路径清单并单测 normalize 函数

**反例**
❌ 错误：OpenAI 鉴权 401 后直接报错，不试 Bearer vs x-api-key  
✅ 正确：鉴权矩阵 × URL 候选笛卡尔探测（有上限）
