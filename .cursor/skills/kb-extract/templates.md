# KB 条目格式与写作规范

每条经验写入 `docs/KB/` 文件时使用以下模板与规范。

## 格式模板

```markdown
## <经验标题>

> 一句话结论：<不超过 30 字，直接说清楚"要做什么"或"不要做什么">

**你会遇到这个问题的场景**
<用 1-3 句话描述：什么技术组合、什么操作、什么阶段容易碰到这个问题。
不提项目名、任务编号、内部模块名。>

**为什么会出错**
<解释根本原因：是语言/框架的设计特性、常见认知误区，还是两个东西组合时的隐蔽冲突。
要让没有参与过这个项目的人能看懂。>

**正确做法**
<可操作的步骤或规则，每条一行。
如果有判断前提（"当 X 时才需要……"），明确写出来。>

**反例（可选）**
<如果错误做法很典型，写出来对比。格式："❌ 错误：…… → ✅ 正确：……">
```

## 写作规范（每条必须遵守）

1. **禁止出现**：任务编号、任务名称、内部模块名、项目专有缩写。
   经验必须脱离原始任务上下文也能独立读懂。

2. **术语必须解释**：
   - 首次出现的技术术语，括号内补充一句说明，例如：
     `queryKey（React Query 用来标识缓存数据的唯一键）`
   - 如果一个术语必须用但难以简短解释，单独加「背景知识」小节。

3. **结论先行**：每条经验第一句话就是结论，背景和原因放后面，
   不要用"在某某情况下，经过某某操作，最终发现……"的叙述顺序。

4. **原因要说透**：不能只写"会导致问题"，要写清楚
   "为什么这个机制/设计会造成这个结果"，让读者能举一反三。

5. **宁少勿滥**：一条写透比三条写浅更有价值。
   如果一个现象背后的原因没想清楚，先跳过，不要强行凑条目。

## 示例（结构参考，非本项目内容）

```markdown
## 缓存键必须包含影响请求结果的全部维度

> 一句话结论：queryKey 缺维度会导致展示过期数据

**你会遇到这个问题的场景**
使用 React Query（服务端状态缓存库）时，列表页按筛选条件请求数据，但切换筛选后 UI 仍显示上一份结果。

**为什么会出错**
React Query 用 queryKey 数组做缓存索引。键相同则命中旧缓存、跳过 refetch。若 queryKey 只含路由不含筛选参数，不同筛选共享同一键，框架无法区分两次请求。

**正确做法**
- queryKey 包含所有影响响应体的输入：筛选、分页、排序、租户 ID 等
- 参数变化时 key 引用变化，自动触发 refetch
- 调试时在 DevTools 中核对 key 是否与预期参数一一对应

**反例**
❌ 错误：`['users']` 不含 filter → ✅ 正确：`['users', { status, page }]`
```
